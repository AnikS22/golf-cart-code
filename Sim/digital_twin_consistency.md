# Digital Twin Consistency ‚Äî the discipline

The Gazebo simulation is a **digital twin** of the real GEM E4 cart. Behavior validated in sim must transfer to the real cart unchanged. The only way that works is if every dimension, sensor pose, sensor spec, ROS topic, TF frame, and control parameter is **identical** between sim and real.

This doc names the consistency contracts and the single source of truth for each.

## Single sources of truth

| Contract | Canonical file | Sim consumes | Real-cart consumes |
|---|---|---|---|
| Vehicle geometry, dynamics, sensor extrinsics, sensor specs | **`Sim/urdf/cart_parameters.xacro`** (xacro property declarations) | Included by `gem_e4_robot.urdf.xacro` (sim) | Included by `Software/autonomy_ws/src/gem_description/urdf/cart_parameters.xacro` ‚Äî **symlink to the canonical file**, do not duplicate |
| CAN protocol / payload structs / msg IDs | **`Software/firmware/common/include/dbw_can_protocol.h`** | Read by `sim_dbw_bridge` (translate to Gazebo) | Compiled into both Teensy firmware AND the real `gem_dbw_bridge` ROS node |
| ROS topic names + message types | Master plan PART B + `cart_parameters.xacro` `TOPIC_*` properties | Sim sensors publish on these | Real sensor drivers publish on these |
| TF tree (frame names + parent-child topology) | URDF (xacros above) | Generated by robot_state_publisher in sim | Generated by robot_state_publisher on cart |
| Control params (PI gains, slew rates, deadbands, governor speeds, watchdog timeouts) | **`Software/autonomy_ws/src/gem_dbw_bridge/config/control_params.yaml`** | sim_dbw_bridge loads | gem_dbw_bridge loads |
| Calibration matrices (K, distortion, LiDAR-cam extrinsic) | `extrinsics/*.yaml` per sensor | sim cheats with ground-truth poses | real cart loads from calibration runs |

## The discipline (rules of the game)

1. **No parameter has two homes.** If you find yourself typing the same number in two files, one of them is wrong. Move it to the canonical source and `<xacro:include>` / `#include` from there.
2. **Topic names are property strings, never string literals.** Refer to `${TOPIC_LIDAR}`, never `/livox/lidar` directly in xacro. Refactor any literal you find.
3. **Frame names match.** `lidar_link`, `zed2i_camera_link`, `imu_link`, etc. ‚Äî same names in sim URDF, real URDF, and all consumer code.
4. **Units agree.** All angles in radians. All distances in meters. All speeds in m/s. CAN protocol fields document their LSB units explicitly (e.g. `centideg`, `permil`) ‚Äî convert at the boundary.
5. **The DBW protocol is the same in sim and real.** sim_dbw_bridge implements the same state machine (DISENGAGED‚ÜíARMED‚ÜíACTIVE‚ÜíFAULT) and same safety gates as the Teensy firmware. Either both accept a command or both reject it.
6. **Calibration values flow ONE way.** Real-cart calibration ‚Üí update `cart_parameters.xacro` ‚Üí sim picks up. Never the reverse (sim values are placeholders; real measurements are ground truth).
7. **Failure modes match.** Sim simulates: E-stop loop opening, Jetson HB loss, EPAS fault, GPS dropout, brake-pedal-pressed override. Same fault flags propagate, same controlled-stop response.

## Acceptance test (run this before committing any cross-cutting change)

If you change anything that touches both sim and real, run:

```bash
# 1. Sim launches, sensors publish on canonical topics, TF tree resolves
ros2 launch gem_sim sim_full.launch.py
ros2 topic list                        # should include /livox/lidar, /zed2i/..., /vn100/imu, etc.
ros2 run tf2_tools view_frames           # tree: map ‚Üí odom ‚Üí base_footprint ‚Üí base_link ‚Üí all sensors

# 2. Real-cart bringup launches with same topic list (firmware on bench, no real cart needed)
ros2 launch gem_bringup gem_full.launch.py use_sim_time:=false
ros2 topic list                        # MUST match sim topic list exactly

# 3. Diff the topic lists ‚Äî should be empty
diff <(ros2 topic list | sort) <(ros2 topic list -t | sort)
```

If diff is non-empty: one of sim or real has a topic the other doesn't. That's a consistency bug. Fix before merging.

## Where consistency typically breaks (and the fix)

| Antipattern | Why it happens | Fix |
|---|---|---|
| Hard-coded mount XYZ in URDF | Quick prototyping | Move to `cart_parameters.xacro` as `EXT_*` property |
| Sim and real publish to slightly different topic names | Forgotten rename | Use `${TOPIC_*}` properties from `cart_parameters.xacro` |
| Different message types in sim vs real | "Just for testing" mock messages | Use stock `sensor_msgs/Image`, `sensor_msgs/PointCloud2`, `sensor_msgs/Imu`, `sensor_msgs/NavSatFix`, `ackermann_msgs/AckermannDriveStamped` everywhere |
| Sim DBW bridge skips state machine | "Just to test motion" | Implement the full DISENGAGED‚ÜíARMED‚ÜíACTIVE‚ÜíFAULT machine on day 1; never skip |
| Calibrated extrinsics on cart, never copied to sim | Time pressure during a test session | Update `cart_parameters.xacro` immediately after calibration; commit before next sim run |
| Updated sim URDF but not the real-cart URDF | Two URDFs exist | Symlink one to the other or share via xacro:include |

## When sim and real legitimately differ

Some things sim does not perfectly replicate. Document each:
- **Wireless RTK**: sim uses Gazebo ground-truth GPS at 10 Hz, no NTRIP / RTCM corrections. Real cart has FLOAT/FIX state transitions; sim is always FIX-equivalent. **Test GPS-degraded behavior on real cart only.**
- **Sensor noise**: sim noise is Gaussian; real LiDAR/cam noise has structure (sun glint, motion blur). Train perception on real bags, not sim.
- **EPAS18 mechanical response**: sim Gazebo plugin steers instantaneously; real EPS column has 50‚Äì200 ms inertia. Tune the steering PI gains on the real cart, not in sim.
- **Vehicle dynamics**: GEM e4's CVT-like behavior, regen braking, and pack voltage sag are not modeled. Validate longitudinal control on real cart.

These are flagged in code comments where they appear.
